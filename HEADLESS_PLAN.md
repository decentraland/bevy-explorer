# Headless bevy-explorer: feasibility & test plan

## 1. Goal & decision to make

Replace hammurabi-headless's model (one Node+Babylon process per scene, ~15 per VM) with **one bevy-explorer engine process + its single `dcl_deno_ipc` V8 sidecar running M scenes concurrently**, orchestrated later by multiplayer-server. The decision this plan enables: **does 1 engine + 1 sidecar at M=15 scenes beat 15 hammurabi processes on memory (primary) with acceptable CPU and tick rate (secondary)?**

Comms is **core scope, not stretch**: the product is hammurabi's runtime model — join each scene's LiveKit room with a pre-minted token, deliver `sceneMessageBus` packets to scene JS, expose player presence to the scene, publish the scene's messages back (hammurabi-headless: connect-adapter.ts:86-124, CommsTransportWrapper.ts:202, scene-context.ts:665-698). M1-M2 run comms-less only to de-risk the engine core first; M3 proves N rooms in one engine; the go/no-go benchmark (M4) includes comms load. Deliverables: a new `src/bin/headless.rs`, the proof sequence M0→M4, and a benchmark with go/no-go thresholds. Orchestrator integration (spawn seam, env protocol) stays out of scope.

## 3. Milestones

### M0 — Run what exists today (zero code, ~1 h)

Tasks: none. Commands (sidecar MUST build first — `init_runtime` panics if `target/{profile}/dcl_deno_ipc` is missing, dcl_deno_ipc/src/lib.rs:106-124; test path pops the `deps` dir, lib.rs:109-112):

```bash
cargo build --package dcl_deno_ipc --release --locked
cargo test --package scene_runner --release --locked -- --test-threads=1   # single-threaded mandatory: global NEW_SCENE_SENDER (dcl_deno_ipc/lib.rs:70-73)
cargo check --bin decentra-bevy --no-default-features --locked             # compile gate for the reduced feature set
```

Caveat on the test command: CI runs `cargo test --all --release --locked -- --test-threads=1` (ci.yml:156-170), where workspace unification pulls the root defaults and builds comms WITH livekit (Cargo.toml:18,26). `--package scene_runner` resolves features from scene_runner's graph only (scene_runner/Cargo.toml:20 declares `comms = { workspace = true }` with no features), producing comms-without-livekit — a combination CI never builds. The cfg-gating in comms looks complete (comms/src/lib.rs:4-5 gates the whole livekit module), so it should compile; if it doesn't, that is a livekit-gating gap in the same remediation bucket as the `cargo check` gate below, not a test regression. CI's exact `--all` invocation remains the fully-proven fallback.

Success: `cyclic_recovery` passes (proves windowless boot + real V8 sidecar handshake + scene load from local ipfs fixtures, test/mod.rs:215-226); `cargo check` exits 0. The no-default-features native path (av with neither `ffmpeg` nor `html`) is an untested compile combination — the wasm build is NOT a precedent for it (wasm builds av WITH `html`, root Cargo.toml:294-295, and `--features livekit` only adds `av/livekit`, Cargo.toml:26, leaving av in exactly the at-risk state). If `cargo check` fails, split the fallback by failing crate: an av failure → fall back to `--features ffmpeg` (restores av's proven native combination); a livekit-gating failure in comms/system_ui/av → fall back to `--features livekit`, noting that shipping `livekit` in the M1-M2 headless bin reintroduces the audio init §4 rules out (CommsPlugin adds LivekitPlugin whenever the feature is on, comms/src/lib.rs:79-80 → kira AudioManager at Startup + per-frame cpal polling) — in that case pull the M3 audio gate forward or explicitly accept it.

De-risks: toolchain/sidecar/socket environment; feature-set compilability. Proves NOT: JS ticks (loop is stubbed).

### M1 — New bin: one scene ticking headless (~1-2 days)

Tasks: create `src/bin/headless.rs` per the spec in §4 (auto-discovered; no `[[bin]]` sections exist). Author one fixture: copy the `empty_scene` triple in `crates/scene_runner/src/test/test_assets/cache/` with `console.log("tick")` in `exports.onUpdate` (runtime calls the export directly, dcl_deno/src/js/mod.rs:322). Fixture authoring rules (load-bearing): set the scene JSON's `"id"` field to the exact hash used as the PortableScenes key — the loader takes id from the JSON body over the requested hash (`definition_json.id.unwrap_or_else(id_fn)`, ipfs/src/lib.rs:152) and portability resolves via `portable_scenes.get(&definition.id)` (initialize_scene.rs:294), so the stock fixture's `"id": "EmptySceneTest"` would silently load with `is_portable = false` (initialize_scene.rs:401). Alternatively delete the `"id"` field so the loader derives it from the file name (ipfs/src/lib.rs:175-187). Also give the fixture a unique `display.title` — portables log the title, and it falls back to `"???"` without one (initialize_scene.rs:373-376). Add a supervisor system modeled on impost's `check_done` (impost.rs:317-375): exit 0 when `tick_number >= 100` (lib.rs:983); exit 1 on `broken` (lib.rs:902,957), hang (`in_flight && now - last_sent > 10s`, renderer_context.rs:122-125), drained `AppError` events (default handler only reaches toasts, lib.rs:1156-1160), or wall-clock timeout. Additionally assert the spawned `RendererSceneContext.is_portable == true` so a pin-key/id mismatch fails loudly instead of silently downgrading the test to the non-portable path.

```bash
cargo build --package dcl_deno_ipc --release
cargo build --bin headless --release --no-default-features   # flags from M0
./target/release/headless --scene <fixture-hash> ...
```

Success (all external): stdout shows `[<fixture display.title> <ts>] tick` at ~30/s — portable scenes log the title, never the parcel (renderer_context.rs:352-358), and the supervisor gates on the unique tick tag in the message payload, not the prefix; exit 0 in <10 s; no window, no panic; clean SIGTERM (sidecar exits with the socket, dcl_deno_ipc/src/main.rs:62-80).

De-risks: the REAL `SceneLoopSchedule` running headless (tests never ran it live); 30 Hz SpinSleeper pacing without a monitor (lib.rs:341-357 fallback); full engine↔sidecar CRDT round-trip per tick; the PortableScenes pin path end-to-end (key ↔ definition.id ↔ is_portable); the exact plugin/resource inventory — any missing resource panics loudly at 1-scene scale.

### M2 — N scenes concurrently in one engine (~2-3 days) — the multi-scene proof

Tasks: extend M1 to insert N PortableScenes entries at startup via the public API — `PortableScenes::insert(hash, source)` (initialize_scene.rs:763) or `PortableScenes::extend` (initialize_scene.rs:792), which also maintain the by_parent index (the `by_hash` field itself is private, initialize_scene.rs:732-736). Urn form: `pid = "urn:decentraland:entity:<hash>?=&baseUrl=<content>/contents/"` (restricted_actions/lib.rs:702-705; resolved via `load_urn`, initialize_scene.rs:167). Do NOT use bare `LoadSceneEvent` — lifecycle despawns scenes not in ScenePointers/PortableScenes/CurrentImposterScene (initialize_scene.rs:1609-1621). Set `scene_threads: N` and `SceneLoadDistance{0,0,0}`. Phase A: N=15 fixture copies, each with a unique JSON `"id"` equal to its pin key, a unique `display.title`, a unique per-tick tag, and a **distinct nonzero base parcel** — with the player pinned OutOfWorld (§4), scene priority is distance² (lib.rs:461-465), so distinct parcels give each scene a distinct priority and the aging quotient actually reorders the queue; identical parcels would collapse every priority to the same value and the stable sort would starve the same tail scenes deterministically (lib.rs:476-479). Phase B: point urns at real remote content — CI conformance realm parcels (`https://decentraland.github.io/scene-explorer-tests/scene-explorer-tests`, ci.yml:83) and/or a Genesis City parcel with GLTFs; drop the `assets_root` override, set a real `starting_realm` so `CurrentRealm`/PbRealmInfo populate (initialize_scene.rs:437-466).

Success: all N per-scene stdout tags advancing; supervisor exits 0 only when every scene reaches tick ≥ 100. Per-scene tick counts within ~2x after 30 s proves the frame budget suffices at N — read skew as budget saturation, and note it is only meaningful because the distinct-parcel setup makes aging real (uniform counts must not be attributed to the aging mechanism in general: at equal priorities it is a no-op). Phase B: a production GLTF scene ticks with `GLTF_LOADING` blocked-state clearing (gltf_container.rs:1082-1092) — first real-content headless run of the CPU-side gltf/rapier path (gltf_container.rs:151-427; mesh_collider.rs:177-229). Log RSS of engine+sidecar at N=1/5/15 as an early signal.

De-risks: PortableScenes pinning headless (now genuinely exercised — M1's is_portable assertion carries over per scene); `scene_threads` throttle at N≥15; scheduling fairness under saturation; the shared 1000-slot response channel under load (`try_send().unwrap()`, dcl_deno_ipc/src/lib.rs:241; frame-budget drain lib.rs:1048-1050); one-sidecar-many-isolates model; remote asset fetch/cache; whether real scenes hang on unhandled RPCs (RestrictedActionsPlugin is included per §4, so `getUserData` etc. are answered — restricted_actions/lib.rs:1078).

### M3 — Comms: one engine in N scene rooms (core, ~1 week)

The multiplayer-server fit lives or dies here. Hammurabi's model per scene: connect to the scene's LiveKit room with the pre-minted bearer token (auth is the token in the adapter URL, not the worker identity — connect-transport.ts:6-8), route `Scene` packets to the scene VM filtered by sceneId (scene-context.ts:674-676), feed participant presence to the scene, publish scene messages back. Bevy must do this for N scenes in one process.

**Already works in bevy (verified against code):**
- Adapter injection: `AdapterManager::connect(&adapter)` takes `livekit:wss://host?access_token=T` directly and spawns a transport entity (comms/src/lib.rs:287-312); tag it `SceneRoom(scene_hash)` (mirroring lib.rs:263-265). The client's gatekeeper singleton (`connect_scene_room`, lib.rs:202-271) only acts on `SetCurrentScene` events the headless bin never emits — it stays dormant, no change needed. Profile must be set before connect (livekit/plugin.rs:70-72) — already mandatory in §4.
- Inbound routing is N-room-ready today: messagebus routes by the `scene_id` INSIDE the RFC4 `Scene` packet to per-scene subscription channels (global_crdt.rs:903-920), not by source transport — the same design as hammurabi's sceneId check.
- Authoritative wire semantics exist client-side already: `authoritative_multiplayer` → `NetworkMessageRecipient::AuthServer` (restricted_actions/lib.rs:1425-1427); inbound `authoritative-server` NonPlayer messages reach scenes (global_crdt.rs:751-774).

**Engine changes required (all scoped, verified):**
1. **Outbound routing fix**: `send_scene_messages` broadcasts each scene's packets to EVERY SceneRoom transport (restricted_actions/lib.rs:1429-1435) — correct with the client's single room, cross-room leakage with N. Fix: skip unless `scene_room.0 == ctx.hash`.
2. **Audio gate**: with `livekit` on, CommsPlugin unconditionally adds LivekitPlugin → kira AudioManager + per-frame cpal polling (comms/src/lib.rs:79-80; livekit/plugin.rs:39,135-145) — gate `MicPlugin`/`build_kira_audio_manager` behind a resource/feature and make `audio_track_is_now_subscribed` take `Option<ResMut<LivekitAudioManager>>` (it needs mut for `.play(...)`, track/plugin.rs:423,451).
3. **Suppress self-position broadcast**: CommsPlugin adds BroadcastPositionPlugin unconditionally (comms/src/lib.rs:43); the fake player would announce itself into every room as a ghost avatar. Hammurabi's server never reports position (`reportPosition` is a stub, connect-adapter.ts:98-100). Gate the system behind a resource.
4. **Player state flows, bounded**: `DiscardPlayerUpdates` must be **false** here (M1/M2 set it true) — scenes need join/leave/position, hammurabi feeds these via its avatar system (scene-context.ts:668-672). Growth controls: stale players despawn after 10 s (global_crdt.rs:929-938), but ForeignPlayer scene-id slots never free — the ~401-player lifetime cap (FOREIGN_PLAYER_RANGE, global_crdt.rs:46) is now **per engine across all 15 rooms**; measure in the M4 soak, remediate by freeing slots if it binds.
5. **CLI**: per-scene adapter pairs — `--scene <hash> --adapter <connstr>` repeated (multiplayer-server mints one adapter per scene room; env mirror later).

**Open design question (decide with the multiplayer-server grouping model):** all rooms feed ONE global player space (GlobalCrdtState) and scenes see players by world position. Genesis scenes pinned at their real parcels isolate naturally by position. Worlds all sit near 0,0 — two worlds in one engine would cross-see players. Options: (a) group engines by coordinate space (multi-scene engine for Genesis; one engine per world), or (b) tag ForeignPlayer with its source transport and filter per scene (engine change). (a) is the cheap first answer and still captures the Genesis fleet win.

Success: 2+ pinned scenes each connected to its own room (local comms-gatekeeper flow as in hammurabi's LocalPreview, connect-adapter.ts:11-41, or manual tokens on a dev LiveKit); a real client joins room A → scene A's JS receives messagebus + onEnterScene while scene B receives nothing; scene A's outbound arrives only in room A (fix #1 verified on the wire); 30 min alive with churn in both rooms.

De-risks: everything the orchestrator integration depends on — after this it's measurement and packaging.

### M4 — Benchmark vs hammurabi-headless (~3-4 days) — the go/no-go

**Topologies**: bevy = 2 PIDs (engine + sidecar, discovered via process tree); hammurabi = M × `node dist/cli.js --position=<x,y> --realm=<url>` (hammurabi-headless/src/cli.ts:18-40; quickjs is in-process wasm, no children).

**Parity**: bevy `fps_target: 30` + `scene_threads >= M`; hammurabi `HAMMURABI_FPS=30` (default 30, src/lib/babylon/index.ts:14). Known asymmetry: hammurabi's scene VM loop free-runs up to ~41 Hz (`MIN_FRAME_TIME=24ms`, game-loop.ts:3) while bevy scenes are engine-paced — report CPU **per scene-tick**, not just per second; optionally patch MIN_FRAME_TIME=33ms for a confirmation run.

**Corpus** (identical published scenes, both runners — hammurabi can't read bevy fixtures): tier-1 idle heartbeat scene; tier-2 heavy-GLTF; tier-3 logic-heavy (conformance realm parcels); **tier-4 comms-loaded** — each of the M scenes connected to its own LiveKit room (M3 wiring) with a synthetic peer per room publishing messagebus at 10 Hz + position at 10 Hz (small Node script on `@livekit/rtc-node`, borrowed from hammurabi's own dep), hammurabi measured under identical load. Pin corpus scenes at distinct parcels so the M2 scheduling regime (distance² priorities under pinned OutOfWorld) carries over. Pin entity hashes at bench time; one warm-up run for caches; verify hash-identical loading in logs.

**Metrics** (5 s cadence, Python psutil sampler in `bench/`): PSS (Linux `/proc/<pid>/smaps_rollup` — the decision number; RSS overstates 15 node processes sharing libnode pages), USS, RSS, cumulative CPU, thread count. **Decision is made on Linux**; macOS runs are harness debugging only. Compare steady-state plateaus (mimalloc retains arenas, src/main.rs:11-12; V8 heaps are uncapped, js/mod.rs:150-156).

**Protocol**: for each tier, M ∈ {1, 2, 4, 8, 15}: 120 s warm-up discarded, 10 min steady-state measured, 3 reps, medians. Liveness gate per sample window: every scene's heartbeat within 10 s + supervisor line `{hash, tick_number, broken, blocked, in_flight_age}`; any `broken`/silent scene discards the run. Derive `mem_total(M)` curves, **marginal memory per scene** (slope fit; expectation: bevy bigger intercept, much smaller slope — per-scene fixed cost ≈ 8 MiB thread stack + isolate + 2×256 KiB CRDT tables + ~1 MiB engine `live_entities`, renderer_context.rs:31,167), crossover M, CPU per scene-tick, achieved Hz.

**Leak soak**: 60 min at M=15 tier-3, both runners. Flag: PSS slope > 0.5 MB/min/scene after minute 10; thread count must stay flat (rising = leaked scene threads, consistent with no-`terminate_execution`, dcl_deno/src/lib.rs:25-26).

**Go/no-go thresholds** (Linux, M=15, medians of 3; **verdict = tier-3 AND tier-4** — idle tiers are intermediate signals only, since production workers are always room-connected):

| Gate | Threshold |
|---|---|
| Memory (primary) | PSS(engine+sidecar) ≤ 0.6 × Σ PSS(15 hammurabi workers) |
| Marginal cost | bevy per-scene slope < 0.5 × hammurabi per-process slope |
| CPU | cpu_total ≤ 1.5 × hammurabi AND achieved tick rate ≥ 29 Hz median / ≥ 25 Hz p5 |
| Idle floor | tier-1 M=15 cpu_total < 25% of one core (catches O(CRDT-processors × scenes) idle overhead, update_world/mod.rs:284-335) |
| Stability | soak slope < 0.5 MB/min/scene; zero broken scenes; threads flat |

No-go remediation before abandoning: heap-cap isolates via `v8::CreateParams` (insertion point js/mod.rs:150) + shrink the 8 MiB stacks; profile IPC serialization (write_msg, dcl_deno_ipc/src/lib.rs:313-320) vs loop-pass overhead.

**Timeline: M0-M2 ≈ 1 week; M3 ≈ 1 week; M4 ≈ 3-4 days → go/no-go verdict ≈ 2.5-3 weeks.**

## 4. Binary spec: `src/bin/headless.rs`

**Build** (features are per-package/additive — a bin target cannot subtract deps; two invocations, and the sidecar must sit next to the bin):

```bash
cargo build --release --package dcl_deno_ipc --locked
cargo build --release --bin headless --no-default-features --locked
```

**Features**: `livekit`/`ffmpeg`/`inspect`/`social` all OFF for M1-M2 (root Cargo.toml:17-28); **from M3 on, the product build turns `livekit` back ON** (rooms are the product) paired with the M3 audio gate. With `livekit` on, CommsPlugin unconditionally adds LivekitPlugin (`#[cfg(feature = "livekit")] app.add_plugins(LivekitPlugin);`, comms/src/lib.rs:79-80) → kira AudioManager at Startup + per-frame cpal polling (plugin.rs:39,135-145; mic.rs:80,160-175) — hostile on an audio-less server, and NOT avoidable by dropping a plugin since CommsPlugin is required: for livekit, heavy init is feature-driven through an already-required plugin. Without the feature, `livekit:` adapters log "comms offline" and continue (comms/lib.rs:313-315). **No new cargo feature is required for the M1-M2 default path**: kira/cpal stay linked (hard deps of common/ipfs/comms — type-level only) but never initialize without their plugins; if M0's fallback forces `livekit` on, pull M3's audio gate forward. Render deps can't be un-linked (workspace bevy features, Cargo.toml:89-120) — fine, they never initialize without RenderPlugin/WinitPlugin. Runtime caveat even with everything off: cpal/kira/gilrs are unconditional non-wasm deps (comms/Cargo.toml:55, common/Cargo.toml:28, ipfs/Cargo.toml:21, comms/Cargo.toml:34-35, av/Cargo.toml:28-29; workspace bevy hardwires `bevy_gilrs`, Cargo.toml:92), which dynamically link libasound and libudev on Linux — see risk 8.

**Plugins IN** (TestPlugins pattern, test/mod.rs:63-113 — NOT the impost/DefaultPlugins pattern): TaskPoolPlugin with explicit bounds — note impost's policy (impost.rs:156-182) is byte-identical to the desktop policy (src/lib.rs:541-566): async_compute {min 2, max 8, 25%}, io {min 8, max 8, 25%}, compute {min 2, max 8, 25%}; the only real contrast is with `TaskPoolPlugin::default()`, whose percentage split can spawn more threads on many-core VMs. Copy the impost/desktop policy for M1-M2; if M4's thread-count/memory numbers warrant fewer threads, defining a smaller policy is new tuning (validate IPFS download throughput at N=15 — num_slots work runs on the IO pool). Then: FrameCount/Time/Transform/Diagnostics/States; `ScheduleRunnerPlugin::run_loop(Duration::ZERO)`; `IpfsIoPlugin{ preview, assets_root, starting_realm, content_server_override, num_slots }` (its real fields, ipfs/src/lib.rs:455-461 — see CLI for cache handling); Asset/Mesh/Gltf/Animation/bevy-ScenePlugin (gltf_container needs SceneSpawner/Assets<Gltf>/Assets<AnimationClip>, gltf_container.rs:151,200,426-427); bevy InputPlugin (ButtonInput resources for InputManager param, input_manager/lib.rs:196-203); ConsolePlugin{add_bevy_console:false}; WalletPlugin; CommsPlugin; DuiPlugin (Res<DuiRegistry> panics without it, scene_ui/mod.rs:989); SystemBridgePlugin{bare:true} (skips the NonSend<WinitWindows> settings systems, system_bridge/lib.rs:52-56); SceneRunnerPlugin; SceneBoundPlugin (insert `PreviewMode` BEFORE it — read at build time, scene_material/lib.rs:307-313); MaterialPlugin::<StandardMaterial> + GizmoPlugin (Gizmos param, raycast_result.rs:84); **UtilsPlugin** (despawn_with — without it scene_ui `DespawnWith` markers leak on a long-running server, common/util.rs:30-36, scene_ui/mod.rs:836); **RestrictedActionsPlugin** (only handler for GetUserData/MovePlayer/SendSceneMessages etc., restricted_actions/lib.rs:67-106,1078 — without it real scenes stall on RPCs; needs EngineMovementControl + AvatarMovementInfo resources (restricted_actions/lib.rs:206-207,438-439) and `init_asset::<Nft>()` (Res<Assets<Nft>> at lib.rs:1482); impost ships these too, impost.rs:286-302 — impost's `SystemAudio` event is for its UiCorePlugin, not RestrictedActions, and nothing in this plugin set reads or writes it, so it is omitted). (Alternative from the smoke draft — omit RestrictedActionsPlugin at M1: rejected; whether an unhandled RpcCall hangs the JS promise is UNVERIFIED and cheap to avoid.)

**Manual inits** (test/mod.rs:128-136): `init_asset::<Shader/AnimationClip/Image/StretchUvMaterial/Font>` + `TextPipeline` + `CosmicFontSystem`.

**Plugins OUT**: RenderPlugin/WindowPlugin/WinitPlugin/bevy UiPlugin (proven unnecessary); AVPlayerPlugin (the kira audio-thread source, audio_source.rs:35); SystemUi/Visuals/Avatar/Nft/Collectibles/WorldUi/TextureCamera/ImageProcessing/Social/Imposter/UserInput/InputManager/UiCore/Metrics (no segment.io from servers, src/lib.rs:260,394); AutomaticTestingPlugin (NOT headless — panics on screenshots folder, spawns Windows, automatic_testing/mod.rs:84-86,285-317). Accepted gap: skipping Tween/avatar-attach/AV drops those CRDT components silently (lib.rs:1015-1017) — tweened engine-side state is incomplete; TweenPlugin is a cheap add-back when fidelity matters.

**Startup sequence**: mimalloc global allocator (src/main.rs:11-12) → parse CLI → `dcl_deno_ipc::init_runtime().unwrap()` on the main thread BEFORE App construction (test/mod.rs:116) → plugins → resources: `AppConfig{ graphics: {vsync:false, fps_target: tick_hz}, scene_threads: max(N,4), scene_log_to_console: true }`; PrimaryPlayerRes(Entity::PLACEHOLDER), PermissionManager, InputMap/InputPriorities/CumulativeAxisData, ToolTips, SceneGlobalLight, TimeOfDay, CursorLocks, PreviewMode (also required by connect_preview_server, preview.rs:25), `SceneLoadDistance{0.0,0.0,0.0}`, `DiscardPlayerUpdates(true)` (M1/M2 only — **false from M3**, scenes need player presence; see M3 item 4); events RpcCall/ScrollTargetEvent/PermissionUsed → Startup systems: fake player, wallet, scene pinning → supervisor.

**Multi-scene pinning**: `PortableScenes::insert(hash, PortableSource{ pid: urn-with-baseUrl, parent_scene: None, ens: None, super_user: false })` or `::extend` (initialize_scene.rs:763,792 — the `by_hash` field is private, initialize_scene.rs:732-736). The pin key MUST equal the loaded definition's id (ipfs/src/lib.rs:152; initialize_scene.rs:294) or the scene silently loads non-portable — fixture JSON ids must be set per copy (see M1). Portables are always-required, deferral-exempt, realm-purge-exempt, and per-urn baseUrl lets scenes from different worlds coexist (initialize_scene.rs:223-228). Caveat: every scene's PbRealmInfo comes from the single CurrentRealm (initialize_scene.rs:437-458; lib.rs:799-832). Rejected alternatives: bare LoadSceneEvent (despawned next lifecycle pass), big SceneLoadDistance (loads everything in a disc), manual ScenePointers (still flows through parcels-in-range).

**Fake player** (mandatory): Startup spawn of one entity `(Transform, Visibility, PrimaryUser::default(), PrimaryCamera::default())` (test/mod.rs:190-197) — `process_scene_lifecycle` early-returns without PrimaryUser (initialize_scene.rs:1522-1524); `PrimaryEntities::player()/camera()` panic without the markers (primary_entities.rs:14-20). No Camera3d/Projection needed (fov push early-returns, lib.rs:1186-1188). **OutOfWorld pinning**: process_realm_change inserts OutOfWorld on every CurrentRealm change including the initial one (initialize_scene.rs:1040-1048), and with SceneLoadDistance{0} handle_out_of_world clears it only when the player's parcel is PointerResult::Nothing (restricted_actions/teleport.rs:111-129) — so the state would flip per realm content at parcel 0,0 (Genesis: Exists → stuck OOW; conformance realm with empty 0,0: Nothing → back in-world), silently changing scheduling regimes between M2 phases (in-world, ContainingScene::get includes ALL portables → every scene priority 0.0, lib.rs:594-605; OOW, portables get distance², lib.rs:598-599,461-465). The headless bin pins one state: the supervisor re-inserts OutOfWorld on the fake player every frame, and fixtures/corpus sit at distinct nonzero parcels (M2), so distance² priorities differ per scene and aging is genuinely exercised in every phase. Guest wallet: `wallet.finalize_as_guest()` (or `finalize_as_guest_with_seed`/`Wallet::finalize` for orchestrator-minted identity, wallet/lib.rs:56-84) + `CurrentUserProfile{ profile: Some(guest), is_deployed: true }` — **mandatory**, for two precise reasons: `profile: Some(...)` because with the default `profile: None` (inserted by UserProfilePlugin via CommsPlugin, comms/src/lib.rs:75; profile/mod.rs:58,295-300) scenes' getUserData freezes parcel scenes and queues portable requests forever (restricted_actions/lib.rs:1087-1099) and livekit connects bail (livekit/plugin.rs start_livekit); `is_deployed: true` because the deploy at profile/mod.rs:242-252 fires when profile is Some AND is_deployed is false — it guards against deploying the guest profile you just inserted (the login.rs:617-631 pattern), not against an un-inserted default.

**SIGTERM + sidecar reaping**: signal handler sets AtomicBool → Update system writes AppExit → App drop drops all `SceneThreadHandle` senders → per-scene KillScene over IPC → scenes drop runtimes → socket closes → sidecar exits (dcl_deno_ipc/lib.rs:189-194; engine.rs:301-308; main.rs:62-80). Belt-and-braces: after AppExit wait ≤5 s then SIGKILL the sidecar pid — a scene blocked in JS never drops its sender and orphans the sidecar even past parent SIGKILL (plain Command, no PDEATHSIG, lib.rs:119-124); this needs `init_runtime` to expose the Child handle — the one small library change in the plan. Sidecar death is unrecoverable by design today (next spawn_scene unwraps a closed channel, lib.rs:290-308): let the process die loudly; the orchestrator restarts.

## 5. Known risks & open questions (ranked; resolving milestone)

1. **Multi-scene headless is unproven end-to-end** — tests stub the scene loop; real JS ticks + N concurrent portables have never run headless. → **M1/M2**.
2. **Memory win may not materialize / CPU may regress** — per-scene fixed costs (8 MiB stacks, uncapped isolates, ~1 MiB live_entities engine-side) vs O(loop-passes × CRDT-processors × scenes) idle overhead (update_world/mod.rs:284-335). → **M4** (that's the whole benchmark).
3. **Shared-fate blast radius**: no V8 heap cap (js/mod.rs:150-156) — one leaky scene OOMs the sidecar and kills all M scenes; engine panics on next spawn (dcl_deno_ipc/lib.rs:290-308); shared 1000-slot response channel `try_send().unwrap()` (lib.rs:241) under frame-budget draining (lib.rs:1048-1050). → **M2** (channel under load) + **M4 soak**; mitigation backlog: CreateParams heap caps.
4. **No forced scene termination**: `terminate_execution` never called, VM_HANDLES insert-only, 30 s timeout dead code (dcl_deno/lib.rs:25-26; js/mod.rs:204-207,413-426); broken scenes never restart (lib.rs:437-439). M1-M4 policy: detect + report + process exit; per-scene kill is a follow-up. → detection wired in **M1**; the risk itself is accepted.
5. **`--no-default-features` native compile is unverified** (av with neither `ffmpeg` nor `html` — a combination neither CI nor the wasm build has ever compiled; wasm builds av WITH html, Cargo.toml:294-295). → **M0**; fallback split by failing crate: `--features ffmpeg` for av failures, `--features livekit` only for livekit-gating failures (with the M3 audio gate pulled forward if it ships in the headless bin).
6. **Real scenes stalling on unhandled RPCs** — mitigated by shipping RestrictedActionsPlugin from M1; empirically confirmed on production content in **M2 Phase B**.
7. **Comms model** (now core, M3): the N-room mechanics are verified feasible (inbound already scene-id-routed, global_crdt.rs:903-920; `AdapterManager::connect` takes pre-minted adapters, comms/lib.rs:287-312) with four scoped changes (outbound room matching, audio gate, position-broadcast suppression, DiscardPlayerUpdates off). Remaining genuinely-open items: the **shared global player space across rooms** (cross-see risk for co-located worlds — resolved by grouping engines per coordinate space, or a per-transport player filter); ForeignPlayer id slots never freed — hard ~401-player lifetime cap now per-engine (global_crdt.rs:440 vs :936); profile-cache growth with player updates enabled (the thing DiscardPlayerUpdates existed to avoid, global_crdt.rs:50-56). → **M3** proves mechanics; **M4 soak + tier-4** quantifies growth.
8. **Container/deployment assumptions**: sidecar binary must sit next to the bin (separate cargo invocation — feature divergence between the two builds is a silent hazard, README.md:38); writable cache/config paths via HOME/XDG_DATA_HOME (impost.rs:62-84; ipfs/src/lib.rs:478-491 panics otherwise); AND the binary + identically-linked sidecar require libasound2 and libudev1 shared libraries at runtime even with all audio features off — cpal/kira/gilrs are unconditional non-wasm deps (comms/Cargo.toml:55, common/Cargo.toml:28, ipfs/Cargo.toml:21, av/Cargo.toml:28-29, bevy_gilrs Cargo.toml:92; build docs require libasound2-dev/libudev-dev, README.md:25, ci.yml:206-207), so a minimal container image without them fails at process load. Install them in the benchmark VM/container image; verify with `ldd target/release/headless`. → **M2 Phase B / M4** on the Linux VM.
9. **MaterialPlugin::<StandardMaterial>/GizmoPlugin without RenderPlugin proven only at unit-test scale**, not in a long-lived process. → **M4 soak**.
10. **Benchmark validity**: tick-pacing asymmetry (hammurabi VM ~41 Hz), allocator retention (mimalloc/V8). Connected-player load is covered by **tier-4**; remaining gap is realistic player counts per room (tier-4 uses 1 synthetic peer/room). → documented controls in **M4**.
